# 数据库与表结构调整建议

## 1. 文档目的

本文档用于说明在现有 Java 仓库管理系统基础上，引入“AI 辅助单据识别模块”后，数据库层面需要如何调整，以及哪些表暂时不需要修改。

本文档目标不是一次性大改数据库，而是遵循：

- 主业务表尽量保持稳定
- 新增能力尽量通过扩展表承接
- 第一版优先保证可落地、可回滚、可调试

---

## 2. 当前数据库现状判断

现有系统核心表通常包括：

- `product`
- `customer`
- `stock`
- `inbound_order`
- `inbound_order_item`
- `outbound_order`
- `outbound_order_item`

这些表已经承担正式业务数据，不建议因为 AI 模块第一版而直接重构。

原因如下：

1. 这些表已经支撑现有商品、客户、库存、入库、出库流程  
2. AI 第一版只是“辅助录入”，不是替代正式业务流程  
3. 如果把识别结果直接揉进正式业务表，后续调试和回滚会很麻烦  

结论：

**第一版不建议大改现有核心业务表，而是通过新增“识别记录表 / 草稿表 / 日志表”来承接 AI 识别过程数据。**

---

## 3. 第一版推荐新增的数据表

第一版最建议增加两张表：

1. `ai_recognition_record`
2. `ai_recognition_item`

如果你希望第一版更轻，也可以先只做一张主表，把明细 JSON 存进去。但从长期维护角度，还是推荐主表 + 明细表。

---

## 4. 表一：ai_recognition_record

### 4.1 作用

用于记录一次 AI 识别任务的整体信息，包括：

- 上传了什么文件
- 识别的是什么单据类型
- 识别状态如何
- 识别出的客户/供应商名称
- OCR 原始文本
- 警告信息
- 最终是否被用户确认生成正式单据

这张表的定位是：

**AI 识别任务主表 / 识别草稿主表**

---

### 4.2 推荐字段

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| task_no | varchar(64) | 识别任务编号 |
| doc_type | varchar(32) | 单据类型，第一版固定为 inbound |
| source_file_name | varchar(255) | 原始文件名 |
| source_file_path | varchar(500) | 文件存储路径，可选 |
| recognition_status | varchar(32) | 识别状态，如 pending / success / failed / confirmed |
| supplier_name | varchar(255) | 识别出的供应商名称 |
| raw_text | text | OCR 原始文本 |
| warnings_json | text | 警告信息 JSON |
| result_json | longtext | 结构化识别结果原始 JSON 备份 |
| error_message | varchar(500) | 识别失败原因 |
| confirmed_order_id | bigint | 用户确认后生成的正式入库单 ID |
| created_by | varchar(64) | 创建人 |
| created_time | datetime | 创建时间 |
| updated_time | datetime | 更新时间 |

---

### 4.3 字段设计说明

#### task_no
建议作为识别任务编号，用于前后端和日志排查时快速定位问题。

#### recognition_status
建议至少支持：

- `pending`
- `success`
- `failed`
- `confirmed`
- `cancelled`

#### raw_text
必须保留。  
这是 OCR 调试最重要的依据之一，后续优化识别效果时非常有用。

#### result_json
建议保留完整结构化结果。  
即使后面已经拆分明细表，仍建议保留原始 JSON，便于回溯和排查。

#### confirmed_order_id
当用户确认并正式生成入库单后，可以把正式单据 ID 回写到这里，形成追踪链路。

---

## 5. 表二：ai_recognition_item

### 5.1 作用

用于记录一次识别任务中识别出的每条商品明细。

这张表的定位是：

**AI 识别明细草稿表**

---

### 5.2 推荐字段

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 主键 |
| recognition_record_id | bigint | 关联 ai_recognition_record.id |
| line_no | int | 行号 |
| product_name_text | varchar(255) | 识别出的商品名称原文 |
| matched_product_id | bigint | 匹配到的商品 ID，可为空 |
| matched_product_name | varchar(255) | 匹配到的商品名称，可为空 |
| quantity | decimal(18,2) | 识别出的数量 |
| unit_price | decimal(18,2) | 识别出的单价 |
| amount | decimal(18,2) | 识别出的金额 |
| remark | varchar(255) | 备注 |
| confidence_score | decimal(5,2) | 识别置信度，可选 |
| match_status | varchar(32) | 匹配状态，如 matched / ambiguous / unmatched |
| created_time | datetime | 创建时间 |
| updated_time | datetime | 更新时间 |

---

### 5.3 字段设计说明

#### product_name_text
这里存 OCR / 清洗后得到的商品名称原文，不要被数据库商品名覆盖。

#### matched_product_id
第一版允许为空。  
因为不是所有商品都能自动匹配到数据库中的正式商品。

#### match_status
建议至少支持：

- `matched`
- `ambiguous`
- `unmatched`

这样前端可以根据状态高亮提示用户。

---

## 6. 第一版是否需要修改现有业务表

### 6.1 inbound_order 是否需要加字段
第一版**不强制需要**。

如果你想在正式入库单中保留“来源于 AI 识别”的信息，可以加一个可选字段：

| 字段名 | 类型 | 说明 |
|---|---|---|
| source_type | varchar(32) | 来源类型，如 manual / ai_assist |

也可以不加，先通过 `ai_recognition_record.confirmed_order_id` 反查。

推荐做法：

**第一版先不改 inbound_order，优先通过 AI 识别记录表关联正式单据。**

---

### 6.2 inbound_order_item 是否需要加字段
第一版**不建议改**。

因为正式入库明细仍然是用户确认后的业务数据，不建议混入 AI 中间状态字段。

---

### 6.3 product 表是否需要新增“别名”字段
第一版不建议直接加。

如果后面商品名称匹配问题比较多，可以考虑新增：

- `product_alias`
- 或独立 `product_alias` 表

但第一版先不扩表，优先让前端人工确认。

---

## 7. 文件存储策略建议

AI 识别通常涉及上传文件，因此需要先定一个策略。

### 第一版推荐
文件保存在本地目录，数据库里只存：

- 原始文件名
- 相对路径 / 存储路径

例如：

- 上传目录：`uploads/ai-recognition/`
- 数据库存：`uploads/ai-recognition/2026/03/xxx.pdf`

这样足够支撑第一版开发和测试。

### 暂不建议第一版就做
- 云存储
- 分布式文件系统
- 对象存储签名访问

---

## 8. 索引建议

### ai_recognition_record
建议加索引：

- `task_no`
- `recognition_status`
- `created_time`
- `confirmed_order_id`

### ai_recognition_item
建议加索引：

- `recognition_record_id`
- `matched_product_id`
- `match_status`

---

## 9. 数据保留与清理建议

由于 OCR 原始文本和 JSON 结果会比较大，建议提前考虑清理策略。

### 第一版建议
先全部保留，便于调试和回溯。

### 后续可加
- 只保留最近 3-6 个月的完整 OCR 文本
- 老数据归档
- 清理失败识别的临时文件

---

## 10. 推荐的第一版 SQL 草案方向

### ai_recognition_record
- 新增表
- 主键自增
- task_no 唯一索引
- created_time、updated_time 时间字段

### ai_recognition_item
- 新增表
- recognition_record_id 外键逻辑关联（数据库层可不强制物理外键，业务层保证）
- 行号 line_no 记录识别顺序

---

## 11. 迁移策略建议

第一版数据库修改建议单独放在一个新脚本中，例如：

- `sql/V1_1__create_ai_recognition_tables.sql`

或者如果你目前没有版本化 SQL 目录，至少统一放在：

- `docs/sql/`
- `database/upgrade/`

避免和旧表初始化 SQL 混在一起。

---

## 12. 本阶段结论

第一版 AI 辅助单据识别模块数据库设计原则如下：

1. 不大改现有核心业务表  
2. 新增 AI 识别主表与明细表承接中间结果  
3. OCR 原始文本、识别 JSON、警告信息必须保留  
4. 正式入库数据仍然走原有业务流程  
5. AI 识别记录通过 confirmed_order_id 与正式单据建立关联  

---

## 13. 推荐下一步动作

1. 确认是否采用 `ai_recognition_record` + `ai_recognition_item` 两表方案  
2. 输出第一版建表 SQL  
3. 在 Java 项目中新增对应 Entity / Mapper / XML  
4. 在前端草稿页中按 `match_status` 展示高亮状态
