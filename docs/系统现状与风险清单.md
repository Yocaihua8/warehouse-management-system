# 系统现状与风险清单

> 本文档基于 2026-04-09 的代码与本机联调结果整理。  
> 目标是说明“当前真实可运行状态”和“剩余风险”，避免把已修复问题继续当成阻塞项。

---

## 1. 当前项目判断

- 项目已形成基本业务闭环：登录鉴权 -> 主数据 -> 库存 -> 手工入/出库 -> AI 识别确认 -> 打印/导出
- 当前成熟度：可运行开发版 / 基本业务闭环
- Web、桌面端、Python AI 服务并存，功能覆盖仍有差异

---

## 2. 当前可运行状态（已验证）

### 已验证通过

- 后端可启动，`/system/health` 返回 `overallStatus=UP`（在 AI 服务同时运行时）
- Flyway 迁移链可通过，当前库版本可到 `v9`
- 前端可启动并访问
- Python AI 服务可启动并通过 `/health`
- 首页趋势 SQL 在 `only_full_group_by` 模式下已修复，不再因 `GROUP BY` 报错

### 运行前提

- MySQL：强依赖
- 后端：强依赖 MySQL，默认读 `local` profile
- AI 服务：仅 AI 识别链路依赖，不影响其余业务

---

## 3. 已收敛问题（近期已修复）

1. Flyway 历史脚本 MySQL 兼容写法已统一到可运行状态，旧库可通过修复历史后继续迁移。
2. `ProductServiceImpl` 启动期直接改表逻辑已移除，结构变更回归 Flyway。
3. `WarehouseManagementSystemApplicationTests` 已去除对本地数据库/Flyway 的强耦合。
4. 商品/客户 `status/code` 前后端契约已对齐（前端可编辑字段已能真实落库）。
5. AI 上传口径已收敛为图片（`jpg/jpeg/png`），不再前端放开 PDF。
6. 首页趋势统计 SQL 已兼容 `only_full_group_by`。

---

## 4. 当前剩余高风险项

### 4.1 端口冲突导致“误判为启动失败”

- 后端 `8080`、AI `9000` 都可能因已有进程占用而报启动失败（常见 `WinError 10048` / `Port 8080 already in use`）
- 这类问题是环境冲突，不是代码回归

### 4.2 配置来源仍不够收敛

- `application.yaml` / `application-local.yml` / `docker-compose.yml` 三套来源并存
- 本地手工启动与 Compose 启动默认值不同（会话存储、数据库密码）

### 4.3 AI 历史列表未分页

- 入库/出库 AI 历史接口与页面仍是全量列表模式
- 数据增长后会影响响应与页面体验

### 4.4 Web 与桌面端能力仍有差异

- 后端与桌面端供应商链路更完整
- Web 端供应商模块仍未完全闭环

---

## 5. 运行边界与排障建议

### 启动顺序

`MySQL -> 后端(8080) -> Python AI(9000，可选) -> 前端(5173)`

### 快速健康检查

- 后端：`http://127.0.0.1:8080/system/health`
- AI：`http://127.0.0.1:9000/health`

### 端口冲突先查再启

- `netstat -ano | findstr :8080`
- `netstat -ano | findstr :9000`
