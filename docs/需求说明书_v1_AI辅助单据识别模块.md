# AI 辅助单据识别模块需求说明书 v1

## 1. 文档信息

- 文档名称：AI 辅助单据识别模块需求说明书 v1
- 适用项目：Java 仓库管理系统
- 当前版本：v1.0
- 编写目的：明确第一阶段 AI 辅助入库单识别功能的业务目标、范围、流程、接口边界与验收标准，作为后续设计与开发依据。

---

## 2. 背景说明

当前仓库管理系统已经具备商品、客户、库存、入库单、出库单等核心后端能力，但单据录入仍主要依赖手工填写。对于纸质采购单、送货单、电子 PDF 单据等场景，人工逐项录入存在以下问题：

1. 录入耗时长。
2. 容易出现商品名称、数量、单价录错。
3. 重复劳动多，纸面单据需要再次进入系统。
4. 单据格式不完全统一，手工整理成本高。

为提高录单效率并降低重复劳动，拟在现有 Java 仓库管理系统中引入一个 Python 智能识别服务，先实现“入库单图片/PDF 上传 → OCR 识别 → 结构化抽取 → 草稿回填 → 用户确认后正式保存”的闭环能力。

---

## 3. 项目目标

### 3.1 总体目标

在不破坏现有 Java 仓库管理系统主业务逻辑的前提下，新增一个 AI 辅助录单模块，用于帮助用户从图片或 PDF 中自动提取入库单信息，并将结果以草稿形式回填到系统中，供用户确认后保存。

### 3.2 第一版目标

第一版只完成以下目标：

- 支持在入库单页面上传图片或 PDF。
- 支持 Java 系统调用 Python 识别服务。
- 支持 Python 返回结构化识别结果。
- 支持前端展示可编辑识别草稿。
- 支持用户确认后，调用现有正式入库逻辑保存数据。

---

## 4. 需求范围

### 4.1 本期包含范围

本期功能包含：

1. 入库单页面新增“智能识别导入”入口。
2. 支持上传 JPG / JPEG / PNG / PDF 文件。
3. Java 后端接收上传文件并调用 Python 识别服务。
4. Python 服务完成：
   - OCR 文本识别
   - 原始文本清洗
   - 字段抽取
   - 返回结构化 JSON
5. Java 侧将识别结果转换为“入库草稿对象”。
6. 前端将草稿数据显示为可编辑表单/表格。
7. 用户确认后，继续调用现有正式入库接口。
8. 正式保存时仍沿用原有库存、商品、事务等业务校验逻辑。

### 4.2 本期不包含范围

本期明确不做：

1. 出库单识别。
2. 手写单识别。
3. 自动直接落库。
4. 多种复杂版式统一识别。
5. 智能补货建议。
6. 自然语言问答查询。
7. 多模型切换与复杂模型训练。

## 5. 用户角色与使用场景

### 5.1 用户角色

本模块第一版主要服务以下用户：

- 仓库管理员
- 店员
- 小型商家负责人

### 5.2 典型使用场景

#### 场景 A：纸质采购单拍照录入
用户收到纸质采购单后，拍照上传到系统，系统自动识别商品和数量，用户确认后生成入库单。

#### 场景 B：供应商送货单 PDF 导入
用户收到电子 PDF 送货单，将文件上传到系统，系统自动提取明细，用户确认后生成入库单。

#### 场景 C：识别不完整时人工补录
系统能识别出大部分字段，但部分商品名或数量缺失，用户在草稿界面手动补全后保存。

---

## 6. 核心业务原则

1. AI 只做辅助识别，不直接写入正式业务数据。
2. 所有正式入库操作仍由 Java 业务系统负责。
3. 用户必须能够查看、编辑和确认识别结果。
4. 正式保存前必须重新执行原有业务校验。
5. 识别结果允许失败，系统必须提供手工录入兜底路径。

---

## 7. 功能需求

### 7.1 页面入口需求

在“新增入库单”页面增加以下功能入口：

- 按钮：`智能识别导入`
- 文件上传组件
- 草稿结果展示区域

### 7.2 文件上传需求

系统应支持上传以下文件类型：

- JPG
- JPEG
- PNG
- PDF

校验要求：

- 文件不能为空
- 文件格式必须受支持
- 文件过大时给出提示（大小阈值可在后续配置）

### 7.3 识别处理需求

上传成功后，Java 系统需将文件转发给 Python 识别服务。Python 需完成：

1. OCR 文本提取
2. 原始文本清洗
3. 字段抽取
4. 结果格式化输出

### 7.4 草稿展示需求

Java 接收到识别结果后，应将其转成草稿数据供前端展示，展示字段至少包括：

- 供应商/客户名称（若识别到）
- 商品明细列表
  - 商品名称
  - 数量
  - 单价
  - 备注
- 警告信息
- OCR 原始文本（调试可见，可配置是否展示）

### 7.5 草稿编辑需求

用户应能够：

- 修改识别到的供应商/客户名称
- 修改商品名称
- 修改数量
- 修改单价
- 删除错误行
- 补充遗漏行
- 手工修正备注

### 7.6 正式保存需求

用户点击“确认生成入库单”后，系统需：

1. 将草稿数据转换为正式入库请求对象
2. 执行现有入库业务校验
3. 保存入库主单与明细
4. 更新库存
5. 返回保存结果

---

## 8. 业务流程

### 8.1 主流程

```text
用户进入新增入库单页面
    ↓
点击“智能识别导入”
    ↓
上传图片 / PDF
    ↓
Java 后端接收文件
    ↓
Java 调用 Python 识别服务
    ↓
Python 执行 OCR + 清洗 + 字段抽取
    ↓
Python 返回结构化结果
    ↓
Java 转换为识别草稿
    ↓
前端展示草稿
    ↓
用户检查并修改
    ↓
点击“确认生成入库单”
    ↓
Java 调用现有正式入库逻辑
    ↓
保存成功
```

### 8.2 异常流程

#### 文件格式非法
```text
用户上传文件
    ↓
系统校验格式失败
    ↓
提示“仅支持 JPG、PNG、PDF 文件”
```

#### Python 识别失败
```text
Java 调 Python
    ↓
Python 返回失败
    ↓
提示“识别失败，请重试或改用手工录入”
```

#### 识别部分缺失
```text
识别完成
    ↓
部分字段缺失或不确定
    ↓
页面高亮提醒
    ↓
用户补全后继续保存
```

#### 正式保存失败
```text
用户点击确认生成入库单
    ↓
Java 执行业务校验
    ↓
校验失败
    ↓
提示错误，不保存
```

## 9. 交互需求

### 9.1 页面状态

页面至少应具备以下状态：

1. 初始状态：未上传文件，允许手工录入。
2. 识别中状态：显示加载中，避免重复提交。
3. 识别完成状态：显示可编辑草稿。
4. 保存中状态：提交正式入库请求。

### 9.2 交互原则

1. 识别结果必须可编辑。
2. 不确定内容必须可见。
3. 失败时必须允许退回手工录入。
4. 用户必须主动点击确认后才允许正式入库。

---

## 10. 数据与接口需求

### 10.1 Python 服务输出建议格式

```json
{
  "success": true,
  "message": "识别成功",
  "data": {
    "supplierName": "某某供应商",
    "items": [
      {
        "productName": "十字螺丝刀",
        "quantity": 20,
        "unitPrice": 12.5,
        "remark": ""
      }
    ],
    "rawText": "OCR原始文本",
    "warnings": []
  }
}
```

### 10.2 Java 草稿对象建议字段

建议新增一个专用草稿对象，而不是直接复用正式入库 DTO。建议字段包括：

- supplierName
- items
  - productName
  - quantity
  - unitPrice
  - remark
  - matchStatus（后续增强可加）
- rawText
- warnings

## 11. 非功能需求

### 11.1 可用性

- 识别失败时要有清晰提示。
- 页面必须支持手工修正。
- 用户不应被迫依赖 AI 才能录单。

### 11.2 稳定性

- Python 服务异常时不能导致 Java 主业务不可用。
- 调用 Python 超时要有明确处理策略。

### 11.3 可维护性

- Java 与 Python 职责边界清晰。
- Python 服务独立部署，避免强耦合。
- 接口输入输出格式固定。

### 11.4 可扩展性

- 后续可扩展为出库单识别。
- 后续可扩展商品模糊匹配与智能补全。

---

## 12. 验收标准

第一版验收通过标准如下：

1. 用户能在入库单页面上传图片/PDF。
2. Java 能成功调用 Python 识别服务。
3. Python 能返回结构化识别结果。
4. 前端能展示并编辑识别草稿。
5. 用户确认后能成功生成正式入库单。
6. 正式保存后库存能按现有逻辑正确更新。
7. 文件错误、识别失败、保存失败时均有明确提示。
8. 失败后仍可切换回手工录入。

---

## 13. 风险与约束

### 13.1 风险

1. OCR 对非标准单据识别率不稳定。
2. 商品名与系统商品库可能无法直接映射。
3. 不同单据版式可能导致字段抽取规则波动。
4. AI 辅助若设计不当，可能误导用户直接相信识别结果。

### 13.2 约束

1. 第一版必须控制范围，只做入库单。
2. 第一版只支持清晰图片与标准 PDF。
3. 第一版不允许直接自动入库。
4. 第一版优先实现闭环，而非追求高阶智能能力。

---

## 14. 后续演进方向

在第一版稳定后，可考虑后续方向：

1. 出库单识别。
2. 商品名模糊匹配。
3. 供应商/客户自动匹配建议。
4. OCR + LLM 文本清洗增强。
5. 低库存分析与补货建议。
6. 自然语言查询与智能报表。

## 15. 结论

本需求说明书定义了 AI 辅助单据识别模块第一版的业务范围、流程、交互与验收标准。第一版目标不是替代人工，而是为现有 Java 仓库管理系统提供一个可控、可校验、可编辑的智能辅助录单能力，为后续更深层的 AI 集成打基础。
