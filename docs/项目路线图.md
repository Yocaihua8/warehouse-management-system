# 项目路线图

## 1. 文档目的

本文档用于明确 Java 仓库管理系统及其 AI 辅助单据识别模块的阶段性目标、优先级、交付物与里程碑，作为项目推进、版本规划与阶段复盘的依据。

---

## 2. 项目总目标

将现有 Java 仓库管理系统逐步建设为一个：

- 核心业务完整可用
- 前后端可操作演示
- 可扩展接入 Python 智能识别服务
- 可用于简历展示、项目答辩与后续真实试点的中小型仓库管理系统

---

## 3. 总体推进原则

### 3.1 主线优先原则
优先完成主业务系统闭环，不让 AI 模块反过来打断主系统收口。

### 3.2 小步快跑原则
每个阶段都必须有明确可见成果，避免长期停留在“设计中”或“开发中”。

### 3.3 稳定优先原则
核心业务以正确性和可维护性为先，AI 模块以“辅助识别、人工确认”为先，不直接替代正式业务逻辑。

### 3.4 版本化推进原则
每一阶段输出都应形成可追踪版本，重要节点更新 README 与变更记录。

---

## 4. 阶段划分总览

| 阶段 | 名称 | 优先级 | 核心目标 | 预期结果 |
|---|---|---:|---|---|
| Phase 1 | Java 主系统前端闭环 | 最高 | 完成核心业务前端页面与主流程打通 | 系统可完整演示 |
| Phase 2 | AI 模块方案与接口预留 | 高 | 确定 AI 功能入口、接口格式与页面预留 | 为 Python 接入做准备 |
| Phase 3 | Python 识别服务原型 | 中高 | 完成文件上传、OCR、结构化返回原型 | Java-Python 通信打通 |
| Phase 4 | 草稿回填与联调 | 中高 | 识别结果回填入库单页面并允许修改 | AI 辅助录单可演示 |
| Phase 5 | 稳定性与展示优化 | 中 | 完善错误处理、日志、文档、截图演示 | 可用于答辩/简历/展示 |
| Phase 6 | 可选增强功能 | 中低 | 继续扩展出库识别、统计、预警等能力 | 项目上限提升 |

---

## 5. 各阶段详细路线

# Phase 1：Java 主系统前端闭环

## 5.1 阶段目标
完成现有 Java 后端的基础管理前端，使系统具备完整业务操作能力。

## 5.2 必做内容

### 页面级
- 登录页
- 首页/仪表盘（可简版）
- 商品管理页
- 客户管理页
- 库存管理页
- 入库单列表页
- 入库单新增页
- 出库单列表页
- 出库单新增页

### 业务级
- 列表查询
- 分页
- 新增/编辑/删除
- 入库单手工录入闭环
- 出库单手工录入闭环

## 5.3 本阶段交付物
- 可运行前端页面
- 可手工完成入库/出库主流程
- README 中新增前端启动与使用说明

## 5.4 验收标准
- 用户可不借助 Postman 完成主要业务操作
- 入库与出库数据能正确写入数据库并联动库存
- 页面基础交互稳定

---

# Phase 2：AI 模块方案与接口预留

## 5.5 阶段目标
在不正式接入 Python 服务的前提下，完成 AI 模块的页面入口预留、数据结构设计和接口约定。

## 5.6 必做内容
- 在入库单新增页中预留“智能识别导入”入口
- 明确 AI 识别草稿对象结构
- 确定 Java 与 Python 的接口输入输出格式
- 完成流程图与需求说明文档

## 5.7 本阶段交付物
- 文档：需求说明书、前期方案、架构设计说明
- 前端：AI 按钮占位或模拟入口
- Java：AI 集成模块目录骨架

## 5.8 验收标准
- 页面已为后续 AI 接入留好位置
- JSON 接口结构已定稿
- 团队/个人对边界理解一致

---

# Phase 3：Python 识别服务原型

## 5.9 阶段目标
搭建独立 Python 服务，实现最小识别原型。

## 5.10 必做内容
- 使用 FastAPI 搭建服务骨架
- 支持图片/PDF 上传
- 打通 OCR 识别流程
- 返回结构化 JSON（可先规则提取）
- Java 能调用 Python 服务并获得响应

## 5.11 本阶段交付物
- Python 服务仓库/子目录
- 识别接口 Demo
- Java 调用接口 Demo

## 5.12 验收标准
- Java 调 Python 成功
- Python 能对标准样本给出结构化结果
- 失败时能返回清晰错误信息

---

# Phase 4：草稿回填与联调

## 5.13 阶段目标
将 Python 识别结果真正嵌入入库单录入流程。

## 5.14 必做内容
- Java 接收 Python 返回结果
- 将结果转换为识别草稿对象
- 前端展示草稿并允许人工修改
- 用户确认后走正式入库逻辑

## 5.15 本阶段交付物
- AI 辅助入库功能第一版
- 识别草稿页面状态管理
- 联调文档与测试样本说明

## 5.16 验收标准
- 用户可通过“上传单据 → 草稿回填 → 修改确认 → 正式入库”完成流程
- AI 结果不会绕过正式业务校验
- 错误提示明确

---

# Phase 5：稳定性与展示优化

## 5.17 阶段目标
完善项目可展示性、可维护性和可复现性。

## 5.18 必做内容
- 增加日志记录
- 增加失败场景处理
- 整理项目截图与流程演示
- 补充 README、版本记录、部署说明
- 整理测试样本与测试结论

## 5.19 本阶段交付物
- 完整 README
- 演示截图/动图
- 版本记录文档
- 测试说明文档

## 5.20 验收标准
- 新人可按文档跑起项目
- 面试/答辩时可完整演示主流程
- GitHub 仓库结构清晰

---

# Phase 6：可选增强功能

## 5.21 可选方向
在主系统收口并稳定后，再选择性扩展：

- 出库单智能识别
- 商品名称智能匹配
- 客户名称模糊匹配
- 低库存预警页面
- 销售与入库统计图表
- 导出 Excel
- 登录鉴权与权限控制
- 操作日志
- Docker 部署

## 5.22 原则
这些能力属于“增强层”，不能影响主系统的完成度和稳定性。

## 6. 版本规划建议

| 版本 | 目标 |
|---|---|
| v1.0.0 | Java 仓库管理系统基础后端完成 |
| v1.1.0 | 基础前端闭环完成 |
| v1.2.0 | AI 识别入口预留与文档完成 |
| v1.3.0 | Python 识别服务原型完成 |
| v1.4.0 | AI 草稿回填流程联调完成 |
| v1.5.0 | 文档、测试、展示优化完成 |

---

## 7. 当前建议优先级

### 当前必须优先做
1. 入库单前端页面完成
2. 出库单前端页面完成
3. 入库单手工录入流程打通
4. AI 模块接口与草稿对象设计完成

### 当前可以同步做，但不要喧宾夺主
1. Python OCR 样本收集
2. Python 服务接口草案
3. OCR 效果小规模预研

### 当前不要急着做
1. 手写识别
2. 大模型增强解析
3. 出库单 AI 识别
4. 智能问答
5. 补货预测

---

## 8. 风险与应对

| 风险 | 说明 | 应对方式 |
|---|---|---|
| 前端未收口就接 AI | 造成流程返工 | 先完成手工入库页面 |
| OCR 识别不稳定 | 单据格式复杂 | 第一版只支持标准样本 |
| 商品无法自动匹配 | 名称别名多 | 先人工确认，不自动落库 |
| Java/Python 接口反复修改 | 边做边改 | 先定 JSON 契约 |
| 范围膨胀 | 同时想做太多 | 严格限制第一版功能范围 |

---

## 9. 阶段复盘建议

每完成一个阶段，至少复盘以下问题：

- 本阶段目标是否全部完成
- 是否有范围外功能侵入
- 是否形成可演示成果
- 文档是否同步更新
- GitHub 是否按规范提交与合并

---

## 10. 总结

本项目推进应坚持“Java 主系统先闭环、Python 智能模块后接入”的路线，先形成完整可操作的管理系统，再逐步引入 AI 辅助能力。第一版 AI 功能以“辅助识别、人工确认、正式保存”闭环为目标，确保项目稳步演进，而不是在技术栈上盲目扩张。
