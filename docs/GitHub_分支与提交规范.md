# GitHub 分支与提交规范

## 1. 文档目的

本文档用于规范 Java 仓库管理系统及 AI 辅助识别模块后续开发过程中的 GitHub 分支管理、提交信息、版本发布与协作方式，避免代码混乱、提交不可追溯和版本失控。

---

## 2. 适用范围

适用于以下仓库开发内容：

- Java 仓库管理系统主项目
- 前端开发分支
- Python AI 辅助识别服务
- 文档、架构设计与测试样本管理

---

## 3. 分支策略

### 3.1 主分支说明

建议至少维护以下两个长期分支：

#### `main`
- 放稳定版本。
- 只保留已验证、可运行、适合展示或发布的代码。
- 不直接在 `main` 上开发日常功能。

#### `dev`
- 作为日常开发主分支。
- 所有新功能、修复、重构先合入 `dev`。
- 当 `dev` 达到阶段稳定后，再合并到 `main`。

### 3.2 功能分支说明

每个独立功能应从 `dev` 拉出单独功能分支进行开发，开发完成后再合并回 `dev`。

命名建议：

- `feature/frontend-inbound-page`
- `feature/ai-recognition-api`
- `feature/python-ocr-service`
- `feature/ai-draft-fill`
- `feature/product-page-refactor`

### 3.3 修复分支说明

针对问题修复可使用：

- `fix/inbound-form-submit-bug`
- `fix/python-timeout-handle`

### 3.4 文档分支说明

针对仅文档更新可使用：

- `docs/requirement-spec-v1`
- `docs/project-readme-update`

---

## 4. 推荐开发流程

### 4.1 新功能开发流程

```text
从 dev 拉取最新代码
    ↓
创建功能分支
    ↓
在功能分支开发与自测
    ↓
提交代码
    ↓
功能完成后合并回 dev
    ↓
dev 稳定后合并回 main
```

### 4.2 实际操作建议

1. 开发前先切到 `dev` 并拉取最新代码。
2. 从 `dev` 新建功能分支。
3. 单个分支只处理一个相对独立的目标。
4. 功能完成并测试通过后，再合并到 `dev`。
5. 不在一个分支中同时混入前端、后端、Python、文档等大量无关修改。

---

## 5. 命名规范

### 5.1 分支命名规范

统一格式：

```text
<type>/<short-description>
```

常见 type：

- `feature/`：新功能
- `fix/`：问题修复
- `refactor/`：代码重构
- `docs/`：文档更新
- `test/`：测试相关
- `chore/`：杂项维护

示例：

- `feature/inbound-ai-entry`
- `feature/python-recognize-api`
- `fix/stock-update-check`
- `refactor/ai-service-layer`
- `docs/readme-architecture`

### 5.2 标签与版本命名规范

版本发布建议使用语义化版本号：

- `v1.0.0`
- `v1.1.0`
- `v1.2.0`

推荐含义：

- 主版本：重大阶段变化
- 次版本：新增功能
- 修订版本：问题修复

## 6. 提交信息规范

### 6.1 基本原则

提交信息必须说明“做了什么”，避免使用无意义描述。

不推荐：

- `update`
- `继续`
- `修改一下`
- `bugfix`
- `test`

推荐使用以下前缀：

- `feat:` 新功能
- `fix:` 修复问题
- `refactor:` 重构
- `docs:` 文档更新
- `test:` 测试相关
- `chore:` 维护类改动

### 6.2 示例

#### 新功能
- `feat: 新增入库单页面基础表单`
- `feat: 增加入库单智能识别导入入口`
- `feat: 新增Python OCR识别服务原型接口`

#### 修复
- `fix: 修复入库单明细数量为空时报错问题`
- `fix: 修复AI识别结果回填后单价丢失问题`

#### 重构
- `refactor: 抽离AI识别服务调用逻辑`
- `refactor: 调整入库草稿对象结构`

#### 文档
- `docs: 新增AI辅助单据识别模块需求说明书`
- `docs: 更新项目版本路线与架构说明`

## 7. 合并原则

### 7.1 合并到 dev 的条件

功能分支合并到 `dev` 前至少满足：

1. 功能目标已完成。
2. 不存在明显编译错误。
3. 本地基本测试通过。
4. 不包含无关试验代码。
5. 提交记录有意义。

### 7.2 合并到 main 的条件

`dev` 合并到 `main` 前至少满足：

1. 阶段目标已完成。
2. 系统能正常启动。
3. 核心流程能跑通。
4. README 已同步更新。
5. 版本说明已整理。

## 9. README 更新规范

每次进入新的里程碑版本时，README 至少同步更新以下内容：

1. 项目简介
2. 当前功能模块
3. 技术栈
4. 项目结构
5. 启动方式
6. 版本更新记录
7. 后续计划

对于 AI 模块版本，建议额外补充：

- AI 模块说明
- 识别流程图
- 演示截图
- 样例说明

---

## 10. 文档管理建议

建议在项目根目录建立 `docs/` 目录，并按主题放置文档，例如：

```text
docs/
  AI_辅助单据识别模块_前期方案.md
  需求说明书_v1_AI辅助单据识别模块.md
  GitHub_分支与提交规范.md
  项目路线图.md
  架构设计说明.md
```

文档修改也应纳入 Git 管理，避免只在本地临时保存。

---

## 11. 个人开发时的简化建议

如果当前主要是个人开发，可采用简化版策略：

- `main`：稳定展示分支
- `dev`：开发分支
- 功能分支：按需新建

即使暂时没有多人协作，也建议保留这套分支思想，因为它会让后续版本推进更清晰。

---

## 12. 不推荐做法

以下做法应尽量避免：

1. 所有开发都直接推到 `main`。
2. 一个提交混入大量不相关修改。
3. 提交信息无法说明改动内容。
4. 试验代码和正式代码长期混在一起。
5. README 长期不更新。
6. 文档、流程图、规范只停留在聊天记录中，不落地到仓库。

---

## 13. 结论

GitHub 版本管理不是形式问题，而是为了让项目持续可控、可追溯、可展示。对当前仓库管理系统及其后续 AI 模块开发而言，建议以 `main + dev + feature/fix/docs` 的轻量分支策略推进，在每个阶段形成清晰的版本节点、文档记录与提交痕迹，保证项目能从练手代码逐步演进为结构清晰的完整作品。
