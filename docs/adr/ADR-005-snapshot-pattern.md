# ADR-005: 订单明细使用数据快照模式

## 状态
已采纳（2025-06）

## 背景

入库单/出库单明细需要关联商品主数据（名称、规格、单位）。商品主数据会被修改（如更正规格描述）。需要决定：历史订单明细是否跟随主数据变更，还是保留下单时的快照。

## 决策

订单明细在**创建时**写入快照字段，后续主数据变更不影响历史订单的展示：

```java
// inbound_order_item 快照字段
productNameSnapshot     // 商品名称（创建时写入）
specificationSnapshot   // 规格（创建时写入）
unitSnapshot            // 单位（创建时写入）
```

`outbound_order` 主表同样存储 `customer_name_snapshot`，防止客户更名后历史单据信息丢失。

查询订单明细时优先使用快照字段展示，不 JOIN 主数据表获取当前值。

## 后果

**正面影响：**
- 历史单据展示数据稳定，商品改名/改规格不影响已有记录
- 符合财务凭证不可追溯修改的业务要求
- 查询明细无需 JOIN 商品表，性能更优
- 主数据删除后历史单据仍可正常展示

**负面影响 / 权衡：**
- 商品主数据与快照存在两份冗余，需要接受数据不一致的设计
- 如需展示"商品当前状态"，需额外 JOIN 主数据表
- 快照字段需在创建/编辑时显式写入，不能遗漏

## 备选方案

**实时 JOIN 主数据**：查询时始终关联最新主数据，历史订单会随主数据变更而变化。不符合"历史记录不可变"的业务要求，已排除。

**软删除 + 版本号**：维护商品的历史版本，订单明细记录版本号。实现复杂度高，对当前业务规模过度设计。
