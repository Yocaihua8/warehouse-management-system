# AI 辅助单据识别模块前期方案（v1）

## 1. 项目背景

当前 Java 仓库管理系统已经完成了后端核心业务能力，包括：

- 商品管理
- 客户管理
- 库存管理
- 入库单管理
- 出库单管理

下一阶段希望在现有系统基础上，引入 Python 智能辅助能力，先从 **单据识别辅助录入** 入手，减少人工录单成本，提升入库效率。

本方案聚焦于第一版落地路径，不追求一次性做大，而是优先完成一个可演示、可迭代、可扩展的最小可行版本（MVP）。

---

## 2. 项目目标

### 2.1 总体目标

在现有 Java 仓库管理系统中，引入一个独立的 Python 智能识别服务，实现以下能力：

1. 用户在入库单页面上传图片或 PDF 单据。
2. Java 系统将文件发送给 Python 识别服务。
3. Python 服务执行 OCR、文本清洗和字段抽取。
4. Java 系统接收识别结果，并回填为“入库草稿单”。
5. 用户检查和修改识别结果。
6. 用户确认后，再调用原有正式入库逻辑完成保存与库存更新。

### 2.2 第一版目标

第一版只聚焦：

- 入库单识别
- 图片 / PDF 上传
- OCR + 规则抽取
- 识别结果草稿回填
- 用户确认后正式保存

### 2.3 第一版不做

为防止范围失控，第一版暂不包含：

- 手写单据识别
- 出库单识别
- 自动直接落库
- 智能补货建议
- 自然语言问答
- 多单据类型统一识别
- 复杂表格结构智能恢复

## 3. 需求分析

## 3.1 用户角色

第一版主要面向以下角色：

- 仓库管理员
- 店员
- 小型商家负责人

这些用户的共同特点是：

- 经常需要根据纸质单据或电子单据录入系统
- 手工录入效率低
- 容易出现商品名称、数量、单价录错的问题
- 希望系统“先帮忙录”，再由自己核对确认

## 3.2 用户痛点

当前纯手工录入存在以下问题：

1. 录单速度慢，重复劳动多。
2. 单据信息需要再次输入系统，操作繁琐。
3. 商品名称存在简称、别名、错别字等问题，人工录入容易出错。
4. 图片/PDF 单据无法直接利用，系统不能辅助理解内容。

## 3.3 功能需求

### 核心功能需求

1. 用户可在入库单页面点击“智能识别导入”。
2. 用户可上传图片（JPG/PNG）或 PDF 文件。
3. Java 后端可将文件转发给 Python 识别服务。
4. Python 服务能够返回结构化识别结果。
5. 前端能够展示识别结果草稿。
6. 用户能够修改识别错误内容。
7. 用户确认后，系统按原有入库单逻辑正式保存。
8. Java 系统对正式保存数据继续执行原有业务校验和事务控制。

### 非功能需求

1. 识别失败时要有明确提示。
2. 识别结果必须可编辑。
3. 不允许 AI 识别结果绕过业务校验直接入库。
4. 模块设计应具备后续扩展到出库单识别的可能。
5. 系统需要保留 OCR 原始文本，便于调试与问题排查。

## 4. 模块边界与职责划分

## 4.1 Java 主系统职责

Java 仍然是核心业务系统，负责：

- 用户登录与权限（后续）
- 商品、客户、库存、入库、出库的正式业务逻辑
- 数据库操作
- 页面展示与交互承接
- 业务校验
- 事务控制
- 最终正式保存单据

一句话概括：

**Java 负责正式业务。**

## 4.2 Python 识别服务职责

Python 作为独立辅助服务，负责：

- 文件接收
- OCR 文字识别
- 文本清洗
- 字段抽取
- 商品/客户名称的初步识别建议
- 返回结构化 JSON 数据

一句话概括：

**Python 负责智能识别。**

## 4.3 用户职责

用户负责：

- 上传单据
- 核对识别结果
- 修改识别错误信息
- 最终确认生成正式入库单

关键原则：

**AI 只做辅助识别，不直接替用户决定正式业务数据。**

## 5. 业务流程设计

## 5.1 主流程

```text
用户进入入库单页面
    ↓
点击“智能识别导入”
    ↓
上传图片 / PDF
    ↓
Java 后端接收文件
    ↓
Java 调用 Python 识别服务
    ↓
Python 执行 OCR → 文本清洗 → 字段抽取
    ↓
Python 返回结构化 JSON
    ↓
Java 将识别结果转换为“入库草稿”
    ↓
前端展示草稿内容
    ↓
用户检查并修改
    ↓
点击“确认生成入库单”
    ↓
Java 调用现有正式入库逻辑
    ↓
保存主单、明细、更新库存
    ↓
返回成功结果
```

## 5.2 异常流程

### 情况 A：上传文件无效

```text
用户上传文件
    ↓
文件为空 / 格式不支持
    ↓
系统提示：请上传 JPG、PNG 或 PDF 文件
```

### 情况 B：Python 识别失败

```text
Java 调用 Python
    ↓
Python 返回失败
    ↓
前端提示：识别失败，请重试或手工录入
```

### 情况 C：识别结果字段不完整

```text
识别结果返回
    ↓
部分字段缺失或不确定
    ↓
页面高亮待确认字段
    ↓
用户手动补全
```

### 情况 D：确认保存时校验失败

```text
用户点击确认生成入库单
    ↓
Java 执行业务校验
    ↓
数据不合法 / 商品不存在 / 数量非法
    ↓
返回错误信息，不保存
```

## 6. 交互逻辑设计

## 6.1 功能位置

第一版建议将功能直接放在：

**入库单新增页面**

原因：

- 单据识别本质上是辅助录入能力，而不是独立业务模块。
- 识别结果最终要落到入库单页面表单中。
- 便于后续自然嵌入“确认保存”流程。

## 6.2 页面新增元素

建议在入库单新增页面中增加以下交互元素：

### 顶部操作区

- 按钮：`智能识别导入`
- 按钮：`手工录入`
- 文件上传控件

### 识别结果区域

- 识别到的供应商 / 客户名称
- 识别到的商品明细表格
  - 商品名称
  - 数量
  - 单价
  - 备注
  - 匹配状态
- 警告提示区
  - 未识别字段
  - 模糊匹配字段
  - 建议人工确认字段

### 底部操作区

- `重新识别`
- `清空草稿`
- `确认生成入库单`

## 6.3 页面状态设计

### 状态 1：初始状态

- 页面为空
- 可直接手工录入
- 可点击“智能识别导入”

### 状态 2：识别中

- 显示“正在识别，请稍候”
- 禁止重复提交文件

### 状态 3：识别完成

- 展示草稿数据
- 用户可逐项修改字段
- 用户可删除错误明细、补充缺失项

### 状态 4：确认保存中

- 调用正式入库接口
- 保存成功后清空草稿或跳转列表页

## 6.4 交互原则

1. 识别结果必须可编辑。
2. 不确定字段必须可见。
3. AI 只是“帮填表”，不是“替用户决定”。
4. 失败时必须允许退回手工录入。

## 7. 技术选型与架构设计

## 7.1 总体架构

采用：

**Java 主系统 + Python 独立识别服务**

两者通过 HTTP REST API 通信。

这样做的原因：

1. 职责分离清晰。
2. 不影响 Java 主业务系统稳定性。
3. 后续可单独升级 Python 识别模块。
4. 扩展出库单识别、智能匹配等能力更方便。

## 7.2 Java 技术栈

沿用现有技术栈：

- Java 17
- Spring Boot
- MyBatis
- MySQL
- Maven

建议在 Java 中新增 AI 集成层，避免将识别逻辑硬塞进原有入库业务代码。

### Java 侧建议新增模块

- `controller`
  - `AiAssistController`
- `service`
  - `AiRecognitionService`
- `service.impl`
  - `AiRecognitionServiceImpl`
- `dto`
  - `AiRecognizeResponseDTO`
  - `AiRecognizedItemDTO`
- `vo`
  - `AiInboundDraftVO`

## 7.3 Python 技术栈

建议第一版使用：

- Python 3.x
- FastAPI
- OCR 工具（后续确定具体方案）
- 规则清洗逻辑

Python 侧职责：

- 接收文件
- OCR 提取文本
- 文本清洗
- 字段抽取
- 返回 JSON

### Python 侧建议目录结构

```text
python-ai-service/
├─ app/
│  ├─ main.py
│  ├─ routers/
│  │  └─ recognize.py
│  ├─ services/
│  │  ├─ ocr_service.py
│  │  └─ parse_service.py
│  ├─ schemas/
│  │  └─ recognize_schema.py
│  └─ utils/
│     └─ file_utils.py
└─ requirements.txt
```

## 7.4 通信方式

采用 HTTP 接口调用。

### 示例接口

`POST /ai/recognize/inbound`

### Python 返回示例

```json
{
  "success": true,
  "message": "识别成功",
  "data": {
    "supplierName": "某某供应商",
    "items": [
      {
        "productName": "十字螺丝刀",
        "quantity": 20,
        "unitPrice": 12.5,
        "remark": ""
      }
    ],
    "rawText": "OCR原始文本",
    "warnings": []
  }
}
```

## 8. 接口草案设计

## 8.1 Java → Python 请求

### 接口地址

`POST /ai/recognize/inbound`

### 请求内容

- 文件：图片 / PDF
- 参数：`docType=inbound`

## 8.2 Python → Java 响应

### 成功响应

```json
{
  "success": true,
  "message": "识别成功",
  "data": {
    "supplierName": "某某供应商",
    "items": [
      {
        "productName": "十字螺丝刀",
        "quantity": 20,
        "unitPrice": 12.5,
        "remark": ""
      }
    ],
    "rawText": "OCR原始文本",
    "warnings": [
      "商品名称可能存在模糊匹配"
    ]
  }
}
```

### 失败响应

```json
{
  "success": false,
  "message": "识别失败，未检测到有效文本",
  "data": null
}
```

## 8.3 Java 侧草稿对象建议

不要直接将 Python 结果映射到正式 `InboundOrderAddDTO`，建议设计独立草稿对象，例如：

- `AiInboundDraftDTO`
- `AiRecognizedItemDTO`

理由：

- 识别结果本质上是“候选数据”，不是正式业务数据。
- 与正式 DTO 分离更有利于后续调试和扩展。
- 可以在草稿对象中保留原始文本、警告信息、匹配状态等字段。

## 9. 开发规范

## 9.1 命名规范

### Java

- Controller：`XxxController`
- Service：`XxxService`
- ServiceImpl：`XxxServiceImpl`
- Mapper：`XxxMapper`
- DTO：`XxxDTO`
- VO：`XxxVO`

### Python

- 文件名：小写下划线
- 类名：大驼峰
- 函数名：小写下划线
- 路由按业务命名，不使用随意缩写

## 9.2 目录规范

### Java

AI 功能独立放置，不直接塞进原有入库单业务文件中。

### Python

按以下逻辑分层：

- router：接口层
- services：业务处理层
- schemas：请求响应对象
- utils：工具函数

## 9.3 接口规范

### Python 返回格式统一为

- `success`
- `message`
- `data`

### Java 继续沿用现有统一响应结构

- `Result<T>`

## 9.4 异常处理规范

### Java

- 正式业务错误继续使用 `BusinessException`
- 调 Python 服务异常需转换为用户可读提示

### Python

- 不返回冗长堆栈给 Java
- 识别失败统一返回结构化错误信息

## 9.5 日志规范

### Java 记录内容

- 文件上传时间
- 调 Python 请求耗时
- Python 返回状态
- 用户确认生成单据结果

### Python 记录内容

- 文件接收信息
- OCR 识别结果摘要
- 字段抽取结果摘要
- 失败原因

## 9.6 测试规范

第一版至少准备以下测试样本：

- 正常清晰单据样本
- OCR 失败样本
- 字段缺失样本
- 商品名称识别不准样本
- 保存校验失败样本

建议采用以下简化策略：

### `main`

- 存放稳定版本

- `feature/frontend-inbound-page`
- `feature/ai-recognition-api`

## 13. 项目结论

本阶段的总体结论是：

在现有 Java 仓库管理系统基础上，引入独立的 Python 智能识别服务是可行且合理的。第一版应严格聚焦入库单场景，通过图片/PDF 上传、OCR 识别、结构化抽取、草稿回填与用户确认保存，完成一个可运行、可演示、可迭代的闭环能力。

项目应采用 **Java 主系统 + Python 独立服务** 的架构，通过 HTTP 接口通信，先完成需求设计与前端闭环，再逐步推进服务原型、联调与测试，避免主系统未收口前过早并行开发造成返工。
