# 架构设计说明

## 1. 文档目的

本文档用于说明 Java 仓库管理系统及其 Python AI 辅助识别模块的总体架构、职责边界、模块划分、通信方式、数据流转与设计原则，为后续开发和重构提供统一依据。

---

## 2. 总体架构目标

本项目采用“主业务系统 + 辅助智能服务”的架构模式：

- Java 系统负责核心业务闭环
- Python 服务负责 OCR 与结构化识别
- 前端负责交互展示与人工确认
- 数据库负责业务数据持久化

项目总体目标是：

- 保持主业务系统稳定
- 将 AI 能力以低耦合方式接入
- 确保识别结果不绕过正式业务校验
- 支持后续平滑扩展更多 AI 能力

---

## 3. 总体架构图（文字版）

```text
前端页面（入库单新增页 / AI识别入口）
        ↓
Java 主业务系统（Spring Boot）
        ├── 商品 / 客户 / 库存 / 入库 / 出库模块
        ├── AI 集成控制层
        ├── AI 集成服务层
        └── MyBatis + MySQL
        ↓
Python 智能识别服务（FastAPI）
        ├── 文件接收
        ├── OCR 识别
        ├── 文本清洗
        └── 字段抽取
```

---

## 4. 技术架构选型

## 4.1 Java 主系统

### 技术栈
- Java 17
- Spring Boot
- MyBatis
- MySQL
- Maven

### 选择原因
- 现有项目已基于此技术栈搭建，适合继续扩展
- 分层清晰，利于维护和业务增强
- 适合承担正式业务逻辑、事务控制和数据库一致性保障

---

## 4.2 Python 智能识别服务

### 技术栈
- Python 3.x
- FastAPI
- OCR 工具（待具体选型）
- 规则抽取 / 文本清洗逻辑

### 选择原因
- Python 更适合做 OCR、文本处理、AI 能力接入
- FastAPI 适合快速搭建 JSON API 服务
- 便于后续扩展模糊匹配、大模型清洗等能力

---

## 4.3 前端

### 定位
- 管理后台型前端
- 承载业务页面与 AI 识别结果交互

### 页面重点
- 入库单录入页
- AI 草稿结果展示区
- 用户确认与修改入口

---

## 5. 分层与职责边界

# 5.1 Java 系统职责

Java 系统是主系统，负责以下内容：

### 用户交互与请求接收
- 接收前端请求
- 接收文件上传请求
- 向前端返回统一结果

### 业务逻辑处理
- 商品管理
- 客户管理
- 库存管理
- 入库单保存
- 出库单保存
- 正式业务校验
- 事务控制

### AI 模块集成
- 调用 Python 识别接口
- 接收识别结果
- 转换识别结果为草稿对象
- 将草稿对象提供给前端展示

### 数据持久化
- 使用 MyBatis 操作 MySQL
- 保存商品、客户、库存、单据等正式业务数据

### 关键原则
Java 只接受“用户确认后的数据”进入正式业务逻辑。

---

# 5.2 Python 系统职责

Python 服务是辅助能力模块，负责：

### 文件处理
- 接收图片 / PDF 文件
- 基础文件校验

### OCR 识别
- 提取单据中的原始文本内容

### 文本清洗
- 去除噪声
- 分段整理
- 提高结构化抽取可读性

### 字段抽取
- 识别供应商/客户名称
- 识别商品名
- 识别数量、单价、备注等
- 返回标准 JSON

### 关键原则
Python 只负责“识别建议”，不直接访问 Java 主库，也不直接修改业务数据。

---

# 5.3 前端职责

前端主要负责：

- 提供上传入口
- 展示识别结果草稿
- 允许用户修改识别结果
- 将最终确认结果提交给 Java 正式保存
- 展示错误提示与状态反馈

---

## 6. 模块划分建议

# 6.1 Java 模块划分

建议在现有系统中新增 AI 集成相关模块，避免将逻辑塞进原有入库 Service。

### 建议新增包结构

```text
com.yocaihua.wms
├── controller
│   └── AiAssistController
├── service
│   └── AiRecognitionService
├── service.impl
│   └── AiRecognitionServiceImpl
├── dto
│   ├── AiRecognizeResponseDTO
│   └── AiRecognizedItemDTO
├── vo
│   └── AiInboundDraftVO
└── config / integration（可选）
```

### 模块说明
- `AiAssistController`：提供 AI 识别相关接口
- `AiRecognitionService`：封装调用 Python 服务逻辑
- `AiRecognizeResponseDTO`：接收 Python 返回的结构化结果
- `AiInboundDraftVO`：返回给前端的草稿对象

---

# 6.2 Python 模块划分

建议 Python 服务保持轻量、清晰、可扩展。

### 建议目录结构

```text
python-ai-service/
├── app.py / main.py
├── routers/
│   └── recognize.py
├── services/
│   ├── ocr_service.py
│   ├── parse_service.py
│   └── file_service.py
├── schemas/
│   ├── request.py
│   └── response.py
```

### 模块说明
- `recognize.py`：提供识别 API 路由
- `ocr_service.py`：OCR 识别封装

## 7. 通信架构设计

# 7.1 通信方式

Java 与 Python 之间采用：

**HTTP REST 接口通信**

原因：
- 简单直接
- 易于调试
- 适合当前项目规模
- 后续可独立部署

---

# 7.2 调用流程

```text
前端上传文件
    ↓
Java 接收文件
    ↓
Java 调用 Python HTTP 接口
    ↓
Python 返回结构化识别结果
    ↓
Java 转换为草稿对象
    ↓
前端展示草稿
    ↓
用户确认后调用正式入库接口
```

---

# 7.3 接口设计原则

### 请求统一
- 文件上传统一使用 multipart/form-data
- 如有需要附带 `docType=inbound`

### 响应统一
Python 返回统一 JSON：

```json
{
  "success": true,
  "message": "识别成功",
  "data": {
    "supplierName": "某某供应商",
    "items": [],
    "rawText": "...",
    "warnings": []
  }
}
```

### 错误统一
即使失败，也应返回可读消息，而不是仅返回堆栈错误。

## 8. 数据流转设计

# 8.1 识别阶段数据流

### 输入
- 图片 / PDF 文件

### Python 处理输出
- 原始 OCR 文本
- 供应商名称
- 商品项列表
- 警告信息

### Java 中间层处理
- 将识别结果转换为“草稿对象”
- 不立即写入数据库

### 前端处理
- 展示草稿
- 允许人工修改

---

# 8.2 正式保存阶段数据流

### 输入
- 用户确认后的草稿表单

### Java 正式处理
- 参数校验
- 商品存在性校验
- 入库单保存
- 明细保存
- 库存增加

### 输出
- 保存成功 / 保存失败结果

---

## 9. 设计原则

# 9.1 低耦合原则
Python 服务作为独立模块存在，不与 Java 数据库直接耦合。

# 9.2 单一职责原则
- Java：正式业务
- Python：识别辅助
- 前端：交互确认

# 9.3 安全原则
AI 结果不得直接修改正式业务数据，必须经过用户确认和 Java 校验。

# 9.4 可扩展原则
第一版先做入库单识别，后续可扩展出库识别、模糊匹配、智能推荐等能力。

# 9.5 稳定优先原则
优先保证主系统稳定，AI 功能作为可选增强入口接入。

---

## 10. 异常与容错设计

# 10.1 Java 侧
需处理：
- Python 服务不可用
- Python 接口超时
- Python 返回数据格式异常
- 用户确认保存时业务校验失败

处理策略：
- 转换为统一错误提示
- 不影响已有手工录入流程
- 保留手工录入兜底方案

---

# 10.2 Python 侧
需处理：
- 文件损坏
- 文件格式不支持
- OCR 失败
- 结构化提取失败

处理策略：
- 返回统一失败响应
- 提供错误消息
- 尽量保留原始文本供调试

---

## 12. 部署思路

# 12.1 开发环境
- Java 服务本地运行
- Python 服务本地运行
- MySQL 本地运行

# 12.2 后续部署方向
可考虑：
- Java 与 Python 分服务部署
- 前端单独部署或与 Java 集成部署
- Python 服务独立端口提供接口

第一版不必过度设计为微服务，只需逻辑独立即可。

## 14. 总结

本项目整体架构采用“Java 主业务系统 + Python 独立识别服务”的分工模式，以低耦合方式将 AI 能力引入现有仓库管理系统。Java 负责正式业务闭环与数据一致性，Python 负责 OCR 与结构化识别，前端负责交互展示与人工确认。该架构既适合当前项目阶段，也为后续功能增强保留了清晰扩展空间。
