# 架构设计说明

> 本文档描述**当前已实现**的系统架构，不包含未来规划内容。
> 最后更新：2026-04

---

## 1. 系统定位

系统从**轻量 WMS（仓库管理系统）**出发，逐步演进为**轻 ERP + WMS 一体化**，覆盖商品管理、客户/供应商管理、入库/出库单据、库存管理和 AI 辅助单据识别。

当前阶段：WMS 核心闭环已完成，前端单据工作台正在向 ERP 录单体验对齐（ERP 单据体系扩展规划见 [frontend-order-pages.md](./features/frontend-order-pages.md)）。

系统采用 **Java 主业务系统 + Python 独立识别服务** 的分工模式，同时提供 Web 前端（Vue 3）和桌面端（JavaFX）两种交互形态。

## 4. 后端分层职责

### Controller 层
- 接收 HTTP 请求，参数校验（`@Valid`）
- 调用 Service，返回统一 `Result<T>` 包装
- 不含业务逻辑

### Service 层
- 所有核心业务逻辑：状态流转、权限二次校验、事务控制
- 关键实现：
  - `InboundOrderServiceImpl`：入库单状态流转、库存增加
  - `OutboundOrderServiceImpl`：出库单状态流转、库存扣减、创建时校验库存
  - `AiRecognitionServiceImpl`：AI 识别流程编排（已拆分为 6 个子服务）
  - `StockFlowService`：所有库存变更的统一入口，每次变更写 `stock_adjust_log`
  - `AbstractOrderServiceSupport`：入库/出库服务公共基类（权限校验/分页归一化/导出工具）

### Mapper 层
- MyBatis 数据访问，注解 + XML 混合模式
- 不含业务判断

### 横切关注点

| 组件 | 作用 |
|------|------|
| `LoginInterceptor` | 每次请求验证 token，设置 `CurrentUserContext`（ThreadLocal） |
| `CurrentUserContext` | 在同一请求线程内保存当前用户身份 |
| `TokenStore` | 会话存储网关，支持 DB / Redis 持久化（`AUTH_SESSION_STORE` 环境变量切换） |
| `PasswordMigrationRunner` | 启动时自动将明文密码迁移为 BCrypt 哈希（幂等） |
| `GlobalExceptionHandler` | 捕获 `BusinessException` 等，统一返回 `Result.error(msg)` |

## 5. 关键设计决策

### 5.1 数据快照（Snapshot Pattern）
入库/出库单明细保存商品名称、规格、单位快照，与主数据解耦。主数据变更不影响历史订单展示。

```java
// inbound_order_item 字段示例
productNameSnapshot    // 商品名称快照
specificationSnapshot  // 规格快照
unitSnapshot           // 单位快照
```

### 5.2 统一库存变更入口（StockFlowService）
所有库存变更（手动调整、入库确认、出库确认、作废回滚）必须经过 `StockFlowService`，不允许直接操作 `stock` 表。`StockFlowService` 负责：
1. 更新 `stock.quantity`
2. 写入 `stock_adjust_log` 审计记录

### 5.3 双层权限控制
- **拦截器层**：`LoginInterceptor.isAdminOnlyRequest()` 按路径+方法判断
- **Service 层**：`ensureAdminPermission()` 二次校验

两层都必须存在，不能只依赖一层。Admin-only 路径清单见 [features/auth.md](./features/auth.md)。

### 5.4 出库单创建时校验库存
出库单在**创建草稿时**即校验库存是否充足，不是在确认时才校验。这防止了用户填好单据后提交确认时才看到报错的体验问题。

### 5.5 AI 结果不直接入库
Python 服务只返回识别建议，Java 后端将结果保存为 `ai_recognition_record` + `ai_recognition_item`，不直接生成正式订单。用户经过人工复核确认后，才调用确认接口创建正式单据。

## 7. Python AI 服务接口

Java 通过 `AiPythonOcrClient` 调用，适配入口在 `OcrAdapterService`。

| 接口 | 方法 | 说明 |
|------|------|------|
| `/ocr/inbound/recognize` | POST multipart/form-data | 识别入库单图片 |
| `/ocr/outbound/recognize` | POST multipart/form-data | 识别出库单图片 |
| `/health` | GET | 健康检查 |

Python 返回标准 JSON：

```json
{
  "success": true,
  "data": {
    "supplierName": "某供应商",
    "items": [
      { "productName": "商品A", "specification": "...", "unit": "箱", "quantity": 10, "unitPrice": 50.0, "amount": 500.0 }
    ],
    "rawText": "...",
    "warnings": []
  }
}
```

## 8. 数据库

12 张核心业务表，详细字段和枚举值见 [database.md](./database.md)。

配置说明：

- 本地手工启动默认使用 DB 会话存储

## 9. 已知技术债务

| 问题 | 位置 | 说明 |
|------|------|------|
| 端口冲突导致误判启动失败 | 本地运行环境 | `8080/9000` 端口被占用时报 `Port already in use`，是环境问题不是代码回归 |
| 配置来源不够收口 | `application*.yml` / `docker-compose.yml` | 本地与 Docker 启动默认值不同，覆盖关系不直观 |
| `OrderItemTable` 与 `OrderDetailItemTable` 重复 | `components/order/` | 两组件均 291 行，功能高度相似，应通过 `editable` prop 合并为一个组件 |
| Web 与桌面端能力有差异 | `frontend/` + `desktop-client/` | 部分功能仅桌面端实现，不能默认视为同一产品面 |
