# 架构设计说明

> 本文档描述**当前已实现**的系统架构，不包含未来规划内容。
> 最后更新：2026-04

---

## 1. 系统定位

面向中小型仓储场景的仓库管理系统（WMS），覆盖商品管理、客户/供应商管理、入库/出库订单、库存管理和 AI 辅助单据识别。

系统采用 **Java 主业务系统 + Python 独立识别服务** 的分工模式，并同时提供 Web 前端（Vue 3）和桌面端（JavaFX）两种交互形态。

当前实际运行形态是：

- Web 端已形成核心业务闭环，但更偏联调入口和备用管理入口
- 桌面端覆盖模块更完整，但同样依赖后端 API
- Web 与桌面端功能覆盖并不完全一致，不能默认视为同一产品面

## 4. 后端分层职责

### Controller 层
- 接收 HTTP 请求，参数校验（`@Valid`）
- 调用 Service，返回统一 `Result<T>` 包装
- 不含业务逻辑

### Service 层
- 所有核心业务逻辑：状态流转、权限二次校验、事务控制
- 关键实现：
  - `InboundOrderServiceImpl`：入库单状态流转、库存增加
  - `OutboundOrderServiceImpl`：出库单状态流转、库存扣减、创建时校验库存
  - `AiRecognitionServiceImpl`：AI 识别流程编排
  - `StockFlowService`：所有库存变更的统一入口，每次变更写 `stock_adjust_log`

### Mapper 层
- MyBatis 数据访问，注解 + XML 混合模式
- 不含业务判断

### 横切关注点
| 组件 | 作用 |
|------|------|
| `LoginInterceptor` | 每次请求验证 token，设置 `CurrentUserContext`（ThreadLocal） |
| `CurrentUserContext` | 在同一请求线程内保存当前用户身份 |
| `TokenStore` | 会话存储网关，支持 DB / Redis 持久化（可配置） |
| `PasswordMigrationRunner` | 启动时自动将明文密码迁移为 BCrypt 哈希（幂等） |
| `GlobalExceptionHandler` | 捕获 `BusinessException` 等，统一返回 `Result.error(msg)` |

## 5. 关键设计决策

### 5.1 数据快照（Snapshot Pattern）
入库/出库单明细保存商品名称、规格、单位快照，与主数据解耦。主数据变更不影响历史订单展示。

```java
// inbound_order_item 字段示例
productNameSnapshot    // 商品名称快照
specificationSnapshot  // 规格快照
unitSnapshot          // 单位快照
```

### 5.2 统一库存变更入口（StockFlowService）
所有库存变更（手动调整、入库确认、出库确认、作废回滚）必须经过 `StockFlowService`，不允许直接操作 `stock` 表。`StockFlowService` 负责：
1. 更新 `stock.quantity`
2. 写入 `stock_adjust_log` 审计记录

### 5.3 双层权限控制
- **拦截器层**：`LoginInterceptor.isAdminOnlyRequest()` 按路径+方法判断
- **Service 层**：`ensureAdminPermission()` 二次校验

两层都必须存在，不能只依赖一层。Admin-only 路径清单见 [features/auth.md](./features/auth.md)。

### 5.4 出库单创建时校验库存
出库单在**创建草稿时**即校验库存是否充足，不是在确认时才校验。这防止了用户填好单据后提交确认时才看到报错的体验问题。

### 5.5 AI 结果不直接入库
Python 服务只返回识别建议，Java 后端将结果保存为 `ai_recognition_record` + `ai_recognition_item`，不直接生成正式订单。用户经过人工复核确认后，才调用确认接口创建正式单据。

## 6. 前端架构

```
src/
├── main.js               全局注册 ElementPlus + Router
├── router/index.js       路由定义 + 全局前置守卫（Token 校验）
├── utils/
│   ├── auth.js           localStorage token/user 工具函数
│   └── request.js        Axios 实例 + 请求拦截（自动附加 token）+ 响应拦截（401 跳登录）
├── api/                  按模块拆分的接口函数（11 个文件，对应 11 个 Controller）
├── layouts/
│   └── MainLayout.vue    整体布局（侧边栏 + 顶栏 + 主内容区）
└── views/                页面组件（按业务模块分目录）
```

**路由守卫逻辑**：
1. 访问 `/login` 且已登录 → 重定向到首页
2. 访问其他页面 → 检查 localStorage token → 调用 `/user/me` 验证 → 未通过跳登录页

## 7. Python AI 服务接口

Java 通过 `AiPythonOcrClient` 调用，适配入口在 `OcrAdapterService`。

| 接口 | 方法 | 说明 |
|------|------|------|
| `/ocr/inbound/recognize` | POST multipart/form-data | 识别入库单图片 |
| `/ocr/outbound/recognize` | POST multipart/form-data | 识别出库单图片 |
| `/health` | GET | 健康检查 |

当前限制：

- Python 侧当前只支持图片识别（jpg/jpeg/png）
- PDF 识别尚未接入

Python 返回标准 JSON：

```json
{
  "success": true,
  "data": {
    "supplierName": "某供应商",
    "items": [
      { "productName": "商品A", "specification": "...", "unit": "箱", "quantity": 10, "unitPrice": 50.0, "amount": 500.0 }
    ],
    "rawText": "...",
    "warnings": []
  }
}
```

## 8. 数据库

12 张核心业务表，详细字段和枚举值见 [database.md](./database.md)。

迁移与初始化：
- `backend/src/main/resources/db/migration/` — Flyway 主迁移目录（新环境主流程）
- `sql/wms.sql` — 完整建表 DDL（离线参考）
- `sql/demo_seed_mvp.sql` — 演示数据
- `sql/V1_x` — 历史脚本，仅参考，不再作为主流程

运行配置边界：

- 主配置默认激活 `local` profile
- 本地默认会话存储为数据库
- `docker-compose.yml` 中后端会切到 Redis 会话模式
- 当前数据库结构管理尚未完全收口，仍存在“Flyway + 历史 SQL + 启动期 DDL”并存的痕迹

## 9. 已知技术债务

| 问题 | 位置 | 说明 |
|------|------|------|
| 配置来源不够收口 | `application.yaml` / `application-local.yml` / `docker-compose.yml` | 本地 profile、Docker 环境变量和示例配置并存，覆盖关系不直观 |
| 端口与进程管理易冲突 | 本地运行环境 | `8080/9000` 端口常因已有进程占用导致误判为“启动失败” |
| AI 历史列表未分页 | `AiRecognitionRecordMapper.xml` + 前端 `views/ai/` | 数据增长后接口和页面性能会受影响 |
| Web 与桌面端能力差异 | `frontend/` + `desktop-client/` | 供应商等模块在 Web 端尚未完全闭环 |
